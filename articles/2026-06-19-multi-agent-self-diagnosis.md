# 多Agent协作自诊断架构：从"救火"到"自治"的四层进化

> 当你的多Agent系统每天需要人工介入3.2次、平均故障恢复要42分钟时，是时候考虑自诊断架构了。

## 背景：多Agent系统的"运维噩梦"

多Agent协作系统上线后，最头疼的不是功能开发，而是**故障排查**。

问题在于：
- **调用链不透明**：Agent A调Agent B，B调C，出了问题你不知道是哪个环节挂了
- **告警风暴**：一个故障触发几十条告警，真正有用的就一两条
- **人工介入频繁**：每天要手动重启、重试、排查3-4次

这套四层自诊断架构的目标很明确：**让系统自己发现问题、定位问题、修复问题**。

## 第一层：统一Trace——跨Agent调用链追踪

### 核心问题

Agent A → Agent B → Agent C，中间某一步失败了，你只能看到"最终结果错误"，但不知道是哪一步出了问题。

### 解决方案

给每一次Agent调用生成一个**全局TraceID**，贯穿整个调用链：

```json
{
  "trace_id": "tr_20260619_abc123",
  "span_id": "span_001",
  "parent_span_id": null,
  "agent": "Agent-A",
  "action": "query_stock_price",
  "status": "success",
  "duration_ms": 1200,
  "children": [
    {
      "span_id": "span_002",
      "parent_span_id": "span_001",
      "agent": "Agent-B",
      "action": "fetch_market_data",
      "status": "error",
      "error": "timeout after 5000ms"
    }
  ]
}
```

### 效果

- **根因定位时间**：从28分钟 → **4分钟**
- 告别"逐个Agent翻日志"的低效排查

## 第二层：心跳健康检测——三态模型

### 核心问题

传统健康检测只有"在线/离线"两种状态，但Agent可能处于"在线但异常"的状态（比如响应慢、返回空结果）。

### 解决方案：三态检测

| 状态 | 条件 | 处理策略 |
|------|------|----------|
| 🟢 健康 | 响应时间 < 阈值 + 返回有效结果 | 正常调度 |
| 🟡 亚健康 | 响应时间超标 或 返回空结果 | 降权调度，触发告警 |
| 🔴 故障 | 连续N次超时 或 抛出异常 | 自动熔断，触发自愈 |

### 关键参数

```python
HEALTH_THRESHOLDS = {
    "response_time_warning": 3000,   # ms
    "response_time_critical": 10000, # ms
    "consecutive_failures": 3,
    "check_interval": 30,            # seconds
    "recovery_check_interval": 60    # seconds
}
```

### 效果

- **误报率**：从23% → **2.1%**
- 告别"Agent好好的但一直在报警"的尴尬

## 第三层：依赖拓扑感知——DAG反向传播

### 核心问题

Agent A依赖Agent B和C，B和C同时故障，你会收到3条告警（A、B、C各一条）。但真正的根因只有B和C，A只是受害者。

### 解决方案：DAG依赖图 + 反向传播

1. **构建依赖拓扑图**：记录Agent之间的依赖关系
2. **故障反向传播**：当B故障时，自动标记A为"受影响"而非"故障"
3. **告警收敛**：只告警根因节点，抑制派生告警

```
依赖关系：
  Agent-A → Agent-B
  Agent-A → Agent-C
  Agent-D → Agent-B

当Agent-B故障时：
  - 根因告警：Agent-B 🔴
  - 派生告警：Agent-A 🟡（受影响），Agent-D 🟡（受影响）
  - 最终输出：1条有效告警，而非3条
```

### 效果

- **告警收敛**：47条告警 → **1条有效告警**
- 运维人员不再被告警淹没

## 第四层：自愈执行——故障模式规则库

### 核心问题

发现问题了，但还是要人工去修复。

### 解决方案：预定义故障模式 + 自动修复

建立**故障模式规则库**，每种故障模式对应一个修复动作：

| 故障模式 | 触发条件 | 自愈动作 |
|----------|----------|----------|
| 超时熔断 | 连续3次超时 | 自动重启Agent，等待60s后重试 |
| 空结果循环 | 连续5次返回空结果 | 切换备用数据源 |
| 内存泄漏 | 内存使用 > 阈值 | 触发Agent重启，释放资源 |
| 依赖不可用 | 上游Agent故障 | 切换降级策略，返回缓存数据 |
| 配置错误 | 启动失败 + 错误码匹配 | 回滚到上一个正常配置 |

### 自愈流程

```python
def self_heal(agent_id, fault_pattern):
    # 1. 记录故障
    log_fault(agent_id, fault_pattern)
    
    # 2. 执行修复动作
    action = FAULT_RULES[fault_pattern]["action"]
    result = execute_action(agent_id, action)
    
    # 3. 验证修复
    if verify_recovery(agent_id):
        log_recovery(agent_id, action)
        return True
    else:
        escalate_to_human(agent_id, fault_pattern)
        return False
```

### 效果

- **自动闭环率**：**87.3%**
- 大部分故障无需人工介入

## 实际效果数据

| 指标 | 优化前 | 优化后 | 提升 |

## 落地建议

### 1. 从Trace开始

先做第一层（统一Trace），投入最小、收益最明显。哪怕只是给每个Agent调用加上trace_id，排查效率就能提升一个量级。

### 2. 健康检测要渐进

不要一上来就搞复杂的三态模型，先从"在线/离线"开始，逐步加入响应时间、空结果等指标。

### 3. 自愈规则要保守

初期只覆盖最明确的故障模式（超时、崩溃），不要试图处理所有边界情况。误自愈比不自愈更危险。

### 4. 保留人工兜底

自愈失败必须有清晰的升级路径，87%自动闭环意味着还有13%需要人工介入。

## 总结

多Agent自诊断架构的核心思想是：**让系统具备自我感知、自我修复的能力**。

四层架构各司其职：
1. **统一Trace**：看清调用链
2. **心跳检测**：识别健康状态
3. **拓扑感知**：收敛告警
4. **自愈执行**：自动修复

从"人工救火"到"系统自治"，不是一步到位的，而是一层一层搭建起来的。先做好Trace和检测，再做收敛和自愈，循序渐进。
