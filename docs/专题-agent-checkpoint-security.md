# Agent 检查点设计与安全防护实战：让 AI Agent 真正靠谱

> 技能写得好不好，不在 Prompt 有多花哨，而在检查点能不能兜住底线。本文分享检查点设计的三个常见坑、可复用模板，以及 Agent 安全防护的实践指南。

---

## 一、为什么需要检查点

AI Agent 能力越来越强，但「能力强」和「靠谱」之间差着一道鸿沟。一个 Agent 可能99次输出都很好，但第100次突然幻觉爆发、风格跑偏、逻辑断裂——而你完全不知道它什么时候会翻车。

**检查点（Checkpoint）** 就是这道鸿沟上的护栏。它不是让 Agent 更聪明，而是让 Agent 犯错时能被拦住。

核心理念：**靠谱比聪明重要**。

---

## 二、检查点设计的三个坑

### 坑1：检查点标准「感觉化」

**症状**：检查点写的是「输出质量要高」「内容要有深度」「风格要一致」这类模糊描述。

**解法**：把检查点改成可验证的结构化标准。

| 感觉化标准（反例） | 结构化标准（正例） |
|---|---|
| 内容要准确 | 所有数字必须与原始数据源比对，误差为0 |
| 风格要一致 | 每段开头必须有引导句，用词难度不超过高中水平 |
| 逻辑要通顺 | 每个结论必须有至少一个论据支撑，禁止跳跃推理 |

### 坑2：检查点重叠矛盾

**解法**：合并重叠项，删除矛盾项。每个检查点覆盖一个且仅一个维度。

### 坑3：缺少负面检查点

**解法**：新增负面拦截点，明确禁区。

```
# 负面检查点示例
- 禁止使用「据说」「有人说」「网上流传」等非正式来源
- 禁止对用户未提及的内容进行推测
- 禁止在同一段落中使用超过3个专业术语（除非有解释）
```

## 三、通用检查点模板

### 模板1：准确性拦截

```yaml
accuracy_checkpoint:
  name: "准确性拦截"
  rules:
    - 所有数字必须与原始数据源逐项比对
    - 禁止编造任何未在源数据中出现的信息
    - 引用内容必须标注来源
  failure_action: "拦截输出，回退到数据源重新查询"
```

### 模板2：风格一致性拦截

```yaml
style_checkpoint:
  name: "风格一致性拦截"
  rules:
    - 全文用词风格统一（正式/口语化二选一）
    - 段落长度保持在合理范围内（3-8句）
    - 标题层级不超过3级
  failure_action: "重写不一致段落"
```

### 模板3：逻辑关联拦截

```yaml
logic_checkpoint:
  name: "逻辑关联拦截"
  rules:
    - 每个结论必须有论据支撑
    - 因果关系必须明确标注
    - 禁止跳跃推理（A → C 必须经过 B）
  failure_action: "补充缺失的推理步骤"
```

## 四、AI Agent 安全防护

### 4.1 主要安全威胁

| 威胁类型 | 描述 | 危害等级 |
|---|---|---|
| **提示注入（Prompt Injection）** | 攻击者在输入中嵌入恶意指令，劫持Agent行为 | 高 |
| **数据泄露（Data Leakage）** | Agent 意外输出训练数据、用户隐私或系统信息 | 高 |
| **权限升级（Privilege Escalation）** | Agent 获得超出预期的系统权限 | 高 |
| **越权访问（Unauthorized Access）** | Agent 访问不属于当前用户的资源 | 中高 |

### 4.2 防护措施

- **输入净化**：对所有外部输入进行过滤
- **输出检查**：在输出前检查是否包含敏感信息
- **最小权限**：只给 Agent 必要的权限
- **审计日志**：记录所有关键操作

---

## 五、落地建议

1. 从最关键的1-2个检查点开始
2. 每个检查点上线后观察效果，再迭代优化
3. 负面检查点往往比正面检查点更有效
4. 安全防护是持续过程，不是一次性任务
